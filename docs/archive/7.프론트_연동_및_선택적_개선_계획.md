# 프론트_연동_및_선택적_개선_계획

## 배경/문제
- 백엔드 리팩토링 및 페이징 도입으로 응답 형태가 변경되어 프론트 영향 가능성이 있다.
- 일부 개선 과제는 선택적으로 남아 있어 우선순위/범위를 정리할 필요가 있다.

## 목표
- 프론트에서 필요한 수정 항목을 명확히 정리한다.
- 선택적 개선 과제를 목록화하고 우선순위를 합의한다.
- 변경으로 인한 기능 회귀를 최소화한다.

## 범위
- 포함
  - 백엔드 변경 사항 중 프론트 영향 항목 정리
  - 프론트 수정 범위 제안
  - 선택적 개선 과제 목록화(우선순위 포함)
- 제외
  - 실제 구현(승인 이후 별도 진행)

## 현재 동작 정리
- 목록/검색/피드 응답이 `PageResponse<T>`로 변경됨
  - `items`, `page`, `size`, `hasNext` 구조
- 검색 쿼리는 빈 값일 때 400 응답
- entries 목록도 페이징 적용됨
- thread feed/search는 entries를 배치 로딩하여 응답 데이터는 동일하나 응답 스키마는 변경됨

## 개선 방향
- 프론트는 목록 응답을 `items` 중심으로 처리하고 페이지네이션 UI를 고려한다.
- 빈 검색 요청은 프론트에서 차단하거나 메시지 표시.
- 선택적 개선 과제는 정책/우선순위를 먼저 합의한 후 구현.

## 단계별 계획
- 1단계: 프론트 영향 분석
  - 변경된 엔드포인트: threads list/feed/search/hidden, entries hidden/search
  - 완료 기준: 프론트 수정 필요 항목 목록 확정
- 2단계: 프론트 수정 계획 수립
  - 목록 응답 `PageResponse<T>` 적용
  - 페이지네이션 또는 무한스크롤 적용 여부 결정
  - 완료 기준: 화면별 수정 범위/작업량 추정
- 3단계: 선택적 개선 과제 합의
  - API 버전/경로 정책
  - 응답 래퍼 통일 여부
  - 성능 최적화(인덱스/조인) 여부
  - 테스트 보강 전략
  - 완료 기준: 우선순위 및 진행 여부 확정
- 4단계: 검증 계획 수립
  - 주요 목록/검색/피드 화면에서 데이터 렌더링 확인
  - 검색 빈값 처리 확인
  - 완료 기준: 검증 체크리스트 확정

## 리스크 및 대응
- 프론트에서 응답 스키마 변경 미반영 시 화면 렌더링 실패
  - 대응: 영향 엔드포인트 우선 수정
- 빈 검색 요청이 400으로 바뀌며 UX 저하 가능
  - 대응: 프론트에서 입력 검증 추가

## 검증 계획
- 목록/검색/피드 화면 정상 렌더링 확인
- 페이지 이동/무한스크롤 동작 확인
- 빈 검색 요청 방지 및 메시지 표시 확인
