# Thredge 작업 계획 문서 (v0)

## 0. 개요
Thredge는 “문서”를 작성하는 앱이 아니라, **하나의 아이디어를 하나의 스레드로 만들고 댓글을 시간 순서로 쌓아가며 사고를 발전**시키는 **완전 개인용(thread-first) 사고 로그 앱**이다.  
Notion 같은 거대 올인원 툴을 지양하고, UI/기능을 최소화해 **생각의 시간성, 당시의 완성된 문장, 맥락(context) 보존**에 집중한다.

---

## 1. 배경(문제 정의)
### 1.1 기존 툴의 불만
- 기능이 과도하게 많아 핵심 행동(사고 발전)이 흐려짐
- “하나의 문서” 안에서 모든 것이 이어져 **스레드 간 경계가 흐릿함**
- 수정/재편집 중심 구조로 인해 **과거의 문장/맥락이 덮어써짐**
- “언제 어떤 생각이었는지” 시간적 정보가 희미해져 회고가 어려움

### 1.2 Thredge가 해결하려는 것
- 아이디어를 **스레드 단위로 명확히 분리**하는 UI
- 댓글을 쌓아가며 사고를 디벨롭할 때 중요한:
  - 댓글이 달린 **시간**
  - 그 시점의 **완성된 문장**
  - 그 순간의 **맥락**
  을 “흐리지 않고” 보존

---

## 2. 제품 정체성 & 핵심 철학
### 2.1 한 문장 정의
**“문서를 쓰지 않고, 생각을 시간 순서의 스레드로 쌓는다.”**

### 2.2 핵심 철학(원칙)
1. **Thread-first**: 아이디어 하나 = 스레드 하나. 스레드끼리 절대 섞이지 않음.
2. **Accumulation over rewriting**: 기본 행동은 “수정”이 아니라 “새 댓글로 추가”.
3. **Editable-but-not-inviting**: 기술적으로는 무제한 수정 가능하지만, 수정 UI는 숨겨져 사용을 유도하지 않음.
4. **No delete, only hide**: 삭제 개념을 없애고 **숨김(archive/hidden)**만 제공.
5. **Context is an asset**: “그때의 문장/맥락”을 사고 자산으로 보존.
6. **Personal-only**: 완전 개인용. 공유/협업 전제 없음 → 솔직한 미완성 사고를 보호.

---

## 3. 기존 앱과의 차이점(차별점)
### 3.1 Notion/문서형 메모와의 차이
- 문서 중심(편집/재구성) ❌ → 스레드 중심(누적) ✅
- 페이지/블록/DB 등 범용 기능 ❌ → 핵심 행동 1~2개 ✅
- 최신 상태 덮어쓰기 ❌ → 시간 순서 사고 로그 ✅

### 3.2 Obsidian/MD 기반 관리와의 차이
- 파일/폴더/링크 구조 중심 ❌
- Thredge는 **UI에서 스레드를 명확히 분리**하고, **댓글 타임라인**을 1급 시민으로 둠
- “문서 내 댓글처럼 붙는 구조”가 아니라 **스레드 단위 공간 + 그 하위에 댓글 누적**을 기본으로 함

### 3.3 협업형 코멘트 도구와의 차이
- Thredge의 댓글은 협업 코멘트가 아니라 **‘사고의 버전/흐름’ 기록**
- 멘션/알림/권한/충돌/사회적 글쓰기 압력 없음

---

## 4. 대상 사용자(가정)
- 기능 많은 올인원 툴에 피로감을 느끼는 사용자
- 아이디어를 한 번에 정리하기보다 **시간을 두고 발전**시키는 타입
- “왜 이렇게 생각이 바뀌었지?”를 자주 되짚는 사용자
- (옵션) 외국어로 사고/기록하는 사용자: 언어 교정 및 제안의 가치를 체감

---

## 5. 핵심 유즈케이스
1. **아이디어 스레드 생성 → 댓글로 사고 발전**
2. 스레드들을 **카테고리(태그)로 분류**하고, 특정 카테고리만 모아 보기
3. 댓글/대댓글로 “그 시점의 문장”을 남기며 사고 흐름 보존
4. AI 분석으로 “해야 할 일”을 추출해 투두/알림 목록에 쌓기
5. AI 요청 모드로 선택한 메모(스레드)들 대상으로 검색/요약/가공 요청 수행
6. 카테고리 미지정 시 AI가 자동 분류
7. 모국어가 아닌 언어로 작성 시 자동 어법 점검 및 개선안 제시

---

## 6. 요구사항(기능) — 빠짐없이 정리
### 6.1 플랫폼/반응형
- 모바일 모드 + 데스크톱 모드 모두 지원
- 핵심 흐름(스레드 탐색/작성/댓글 추가/AI 버튼)은 두 모드에서 동일하게 접근 가능

### 6.2 스레드(메모) 관리
- 스레드 생성/조회/수정(제목 등 최소)
- 스레드 간 UI 분리 명확: “각 아이디어는 독립된 공간”
- 스레드 숨김(archive/hidden) 지원 (삭제 없음)
- 숨김 스레드는 별도 뷰/탭에서 확인 및 복원 가능

### 6.3 카테고리(태그) 시스템
- 각 스레드는 다중 카테고리 소속 가능
- 카테고리는 블로그 태그처럼 쉽게 생성 가능
- 생성된 카테고리는 화면에서 **뚜렷하게 노출**되어:
  - 카테고리 기준 필터링
  - 특정 카테고리의 스레드만 모아 보기
  가 가능해야 함
- 스레드 작성 시 카테고리 명시 가능/불가능
  - 명시하지 않으면 AI가 분류(카테고리 제안 또는 자동 부착)

### 6.4 댓글/대댓글(Entry 시스템)
- 스레드 하위에 댓글 작성 가능
- 댓글에 대댓글(계층 구조) 지원
- 각 댓글/대댓글은:
  - 작성 시각이 명확히 표시
  - 작성 당시의 텍스트/문장이 중요(시간성과 함께 보존)
- 댓글도 **숨김** 가능(삭제 없음)
  - 기본 뷰에서는 숨김 처리
  - “숨겨진 댓글 n개 보기” 토글로 흐름을 깨지 않고 열람 가능

### 6.5 편집 정책(수정 가능하되 유도하지 않음)
- 댓글/대댓글은 **제한 없이 수정 가능**
- 단, 수정 UI는 “기본 행동”이 되지 않도록 숨긴다
  - 예: `···` 메뉴/롱프레스/마우스오버 시 노출
  - 인라인 즉시 편집(더블클릭 등)은 지양
- 수정 시 `updated_at` 같은 최소한의 표시로 “수정됨”만 드러내어 시간성 보존

### 6.6 AI 분석 버튼(행동화)
- 메모/댓글 입력 후 **[AI 분석]** 버튼 제공
- AI가 내용 분석 후:
  - “해야 할 일”로 판단되는 항목은 **투두 리스트**에 추가
  - 일정 알림이 필요하다고 판단되면 **일정 알림 목록**에 추가
- 투두/알림 생성은 “추가” 중심(자동 실행보다는 사용자 확인 흐름 권장)

### 6.7 AI 요청 모드(명령/탐색/가공)
- **[AI 요청]** 버튼을 누르면 “메모 입력창”이 **AI 요청창**으로 전환
- AI 요청은 다음을 포함:
  - 메모(스레드) 검색/질의 (의도 기반)
  - 특정 메모(스레드)들을 선택해 그 집합 대상으로 요청 수행
    - 예: 선택한 스레드 요약, 공통 주제 추출, 미결 질문 찾기 등
- 요청 모드에서 “대상 범위(전체/필터된 목록/선택된 스레드)”가 명확해야 함

### 6.8 자동 언어 감지 및 어법 검사
- 사용자의 **주 언어가 아닌 언어**로 글을 작성하면:
  - 자동으로 어법/문법 체크
  - 틀린 부분에 대해 개선안 제시
- 적용 대상: 최소 스레드 본문/댓글(Entry)
- 원문은 보존하고 개선안은 제안 형태로 제공(자동 치환 지양)

## 7. 정보 구조(IA) & 화면 설계(초안)
### 7.1 핵심 화면(최소)
1. **Thread List(스레드 목록)**
   - 카테고리 필터/검색
   - 최근 활동(마지막 댓글 시각) 표시
   - 숨김 제외가 기본

2. **Thread Detail(스레드 상세)**
   - 스레드 상단: 스레드 본문(최초 메모)
   - 하단: 댓글/대댓글 타임라인(시간순)
   - 입력창: 기본은 “댓글 추가”
   - 버튼: [AI 분석], [AI 요청]
   - 숨김 토글: “숨겨진 댓글 n개”

3. **Hidden/Archive(숨김 모음)**
   - 숨김 스레드 목록
   - 스레드/댓글 복원

4. **To-do List(투두)**
   - AI 분석으로 들어온 항목
   - 체크/완료(최소 기능)

5. **Reminders(일정 알림 목록)**
   - 알림 항목 리스트
   - 일정/알림 트리거는 MVP에선 목록 관리 중심(후속 단계에서 OS 알림 연동)

### 7.2 모바일/데스크톱 레이아웃
- 데스크톱: 좌측 목록 + 우측 상세(2-pane) 가능
- 모바일: 목록 → 상세로 전환, 상단에서 카테고리/검색 접근

## 8. UX 디테일(행동 유도 장치)
### 8.1 “수정하지 않게” 만드는 설계
- 댓글의 기본 인터랙션은 읽기 전용
- 수정은 `···` 메뉴(데스크톱), 롱프레스(모바일)로만 제공
- 단축키(옵션)로 숨은 고급 기능 제공 가능

### 8.2 “삭제 없음”의 표현
- “삭제” 단어 미노출
- 모든 제거 행위는 “숨김”으로 표기
- 휴지통 개념 없음, 대신 Hidden/Archive 뷰 제공

### 8.3 시간성 강조
- 댓글 간 시간 간격을 텍스트로 표시(예: “3일 후 다시 생각함”)
- 수정된 댓글은 ‘수정됨’만 조용히 표시

## 9. 데이터 모델(초안)
> 구현 언어/DB는 미정이므로 논리 모델로만 정의

### 9.1 Thread
- id
- title
- body (초기 메모)
- category_ids[] (다중)
- created_at
- last_activity_at
- updated_at (optional)
- is_hidden (boolean)

### 9.2 Entry (댓글/대댓글)
- id
- thread_id
- parent_entry_id (nullable) — 대댓글 지원
- content
- created_at
- updated_at (optional)
- is_hidden (boolean)
- detected_language (optional)
- ai_annotations (optional)

### 9.3 Category
- id
- name (unique)
- created_at
- display_style (color/icon 등 optional)

### 9.4 TodoItem (AI 분석 결과)
- id
- source (thread_id/entry_id 등 참조)
- text
- created_at
- is_done
- metadata (optional: due_candidate, confidence 등)

### 9.5 ReminderItem (AI 분석 결과)
- id
- source (thread_id/entry_id)
- text
- suggested_time_range (optional)
- created_at
- is_hidden/is_done (optional)
- metadata

## 10. AI 기능 설계(초안)
### 10.1 AI 분석(행동화) 파이프라인
입력: Thread/Entry content (+ thread context)  
출력:
- todo_candidates[]: “해야 할 일” 추출
- reminder_candidates[]: “일정 알림 필요” 추출
- category_suggestions[]: 카테고리 미지정 시 제안/자동부착

추가 원칙:
- AI는 “대신 결정”이 아니라 “제안 + 근거”
- 투두/알림 항목에 “왜 이게 생성됐는지” 최소 근거(짧은 한 줄) 포함 권장

### 10.2 AI 요청 모드(명령형)
- 요청 입력창에서 자연어로:
  - 검색(의미 기반), 요약, 비교, 미결 질문, 사고 변화 지점 탐지 등
- 대상 범위:
  - 전체 스레드
  - 현재 필터된 목록
  - 사용자가 선택한 스레드 집합
- 숨김 처리된 Entry 포함 여부:
  - 기본 포함(사고 자산으로 보기)
  - 옵션으로 “숨김 제외” 제공

### 10.3 언어 감지/어법 검사
- 작성 언어 감지
- 주 언어가 아닌 경우:
  - 문법/어법 문제점 표시
  - 자연스러운 대안 문장 제시
  - 원문 보존(제안 형태)

## 11. MVP 범위 제안(1차 릴리즈)
### 11.1 반드시 포함
- Thread/Entry CRUD(단, 삭제 없음, 숨김만)
- 대댓글(계층) 지원
- 카테고리 생성/부착/필터
- 모바일/데스크톱 반응형
- AI 분석: 투두/알림 후보 생성 + 목록에 추가
- AI 요청 모드: 검색/요약/선택 대상 처리(최소 2~3개의 핵심 명령)
- 외국어 어법 체크(기본 동작)

### 11.2 MVP에서 보류 가능
- 댓글 수정 히스토리(버전 관리)
- OS 네이티브 알림/캘린더 연동(처음엔 “알림 목록”으로 충분)
- 고급 권한/공유 기능(철학상 제외)

## 12. 성공 기준(지표 초안)
- “스레드 생성 → 댓글 추가”까지의 입력 마찰이 낮다
- 사용자가 실제로 **수정보다 댓글 추가를 더 자주** 한다
- 숨김 기능이 “정리”를 돕고, 삭제 욕구를 줄인다
- 카테고리 필터링으로 원하는 스레드를 빠르게 찾는다
- AI 분석으로 생성된 투두/알림이 사용자의 실제 행동에 도움이 된다
- 외국어 작성 시 교정 제안이 “귀찮음”이 아니라 “유용함”으로 인식된다

---

## 13. 오픈 이슈(추후 결정)
- 주 언어 설정 방식(사용자 설정 vs 자동 추정)
- AI 요청 모드에서 제공할 “기본 프리셋 명령” 세트
- 숨김 상태의 기본 정책(AI 분석/요약에 포함 여부 기본값)
- 대댓글의 깊이 제한 여부(무제한 vs 2~3단)
- 오프라인 저장/동기화 전략(개인용이므로 로컬 우선 고려 가능)

---

## 14. 요약
Thredge는 기능을 늘려 경쟁하는 앱이 아니라, **사고의 시간성과 스레드 분리 UI**로 정체성을 갖는 개인용 도구다.  
핵심은 “수정/삭제” 중심이 아닌 “추가/숨김” 중심 흐름이며, AI는 글을 대신 쓰는 존재가 아니라 **사고를 해석하고 행동화를 돕는 관찰자**로 동작한다.
