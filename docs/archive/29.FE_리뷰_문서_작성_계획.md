# FE_리뷰_문서_작성_계획

## 배경/문제
- 사용자 요청은 FE 프로젝트가 60·30·10 + 대비, 시각적 위계/근접성, 8px 그리드/반복, Hick’s Law/CTA 설계, Jakob’s law, F패턴 등 원칙을 만족하는지 검토 문서를 작성하는 것임
- 현재 프로젝트의 FE 구현이 원칙을 얼마나 반영했는지 체계적으로 확인하고 기록할 필요가 있음

## 목표
- 지정된 UX/UI 원칙 각각에 대해 적용 여부와 근거를 명확히 서술한 리뷰 문서를 제공
- 화면/컴포넌트 단위로 관찰 내용을 구조화해 재검증 가능한 상태로 남김

## 범위
- 포함 범위
  - FE 코드/스타일/레이아웃/컴포넌트 구조의 정성적 평가
  - 주요 화면(가능하면 메인 플로우)의 디자인 원칙 적용 여부
- 제외 범위
  - 백엔드/인프라 코드 리뷰
  - 사용자 테스트/정량 UX 실험

## 현재 동작 정리
- FE 소스 위치와 라우트/페이지 구조, 공통 레이아웃/스타일 시스템 확인
- 디자인 토큰, CSS 규칙, 레이아웃 그리드 사용 여부 파악

## 개선 방향
- 원칙별 체크리스트 형태로 정리하여 판단 근거를 명확히 제시
- 적용/미적용/부분 적용으로 구분해 모호함 최소화
- 스크린샷/코드 근거가 필요하면 구체적 파일과 요소를 명시

## 단계별 계획
- 1단계: FE 구조 파악
  - 무엇을 변경하는지: 없음(조사 단계)
  - 완료 기준: 주요 화면/레이아웃 파일 목록과 스타일 시스템 위치 정리
- 2단계: 원칙별 리뷰 수행
  - 무엇을 변경하는지: 없음(평가/기록 단계)
  - 완료 기준: 각 원칙에 대해 적용 여부와 근거가 정리됨
- 3단계: 리뷰 문서 작성
  - 무엇을 변경하는지: 리뷰 문서 초안 작성
  - 완료 기준: 사용자 요청 형식에 맞춘 한국어 리뷰 문서 제공

## 리스크 및 대응
- 리스크: 화면 범위가 넓어 평가가 과도하게 분산될 수 있음
  - 대응: 핵심 플로우 중심으로 우선 평가 후 범위 확장
- 리스크: 스타일 시스템 부재로 원칙 적용 여부 판단이 어려움
  - 대응: 개별 컴포넌트/페이지 사례 기반으로 평가 근거 제시

## 검증 계획
- FE 주요 페이지를 기준으로 원칙별 체크가 누락되지 않았는지 자체 점검
- 리뷰 문서에 파일 경로 또는 화면 근거가 포함되었는지 확인
