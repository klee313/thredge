# 프론트 상태/테스트/번역/성능 정리 계획

## 배경/문제
- HomeFeed/ThreadDetail 상태 훅의 인터페이스가 부분적으로 달라 재사용성이 낮음.
- 핵심 컴포넌트에 대한 간단한 검증 수단(스토리/스냅샷)이 부족함.
- 번역 키가 증가하면서 관리 난이도가 올라감.
- 일부 useMemo 의존성과 캐시 업데이트 로직을 더 정리할 여지가 있음.

## 목표
- 상태 훅을 공통 패턴으로 맞춰 확장성과 일관성을 확보한다.
- 핵심 UI 컴포넌트를 간단히 검증할 수 있는 테스트/스토리를 마련한다.
- 번역 키 구조를 정리해 관리성을 개선한다.
- 성능 관련 중복/불필요 연산을 줄인다.

## 범위
- 포함
  - HomeFeed/ThreadDetail 상태 훅 인터페이스 정렬
  - EntryCard/ThreadEditor 등 핵심 컴포넌트 스냅샷/스토리 추가
  - 번역 키 구조 정리(명명/그룹화)
  - useMemo/useCallback 의존성 정리 또는 캐시 유틸 개선
- 제외
  - UI 전면 리디자인
  - 백엔드 API 변경
  - 대규모 상태관리 교체

## 현재 동작 정리
- 상태 훅의 action/state 구조가 페이지별로 다름.
- 컴포넌트 단위 검증이 수동 테스트에 의존.
- 번역 키는 파일 내에서 증가하며 일부 중복 표현이 존재.
- 캐시 업데이트 유틸은 피드 위주로만 정리됨.

## 개선 방향
- 상태 훅은 `state/actions` 구조와 네이밍을 통일한다.
- 핵심 컴포넌트에 대해 최소한의 스냅샷/스토리를 추가한다.
- 번역 키는 기능별 네임스페이스로 정리한다.
- 불필요한 계산/리렌더를 줄이는 방향으로 개선한다.

## 단계별 계획
- Phase 1: 상태 훅 통합
  - useHomeFeedState/useThreadDetailState 인터페이스 정렬
  - 완료 기준: state/actions 구조 일관화
- Phase 2: 테스트/스토리 추가
  - EntryCard/ThreadEditor 스냅샷 또는 스토리 작성
  - 완료 기준: 핵심 컴포넌트에 최소 검증 추가
- Phase 3: 번역 키 정리
  - home/thread/archive/settings 키 구조 재정리
  - 완료 기준: 중복 키 축소 및 그룹화
- Phase 4: 성능 미세 개선
  - useMemo/useCallback 의존성 정리
  - 캐시 업데이트 유틸 확장
  - 완료 기준: 불필요 연산 제거

## 리스크 및 대응
- 번역 키 변경으로 인한 UI 텍스트 누락 가능
  - 대응: 변경 후 페이지별 수동 확인
- 테스트 도입 시 환경 설정 필요
  - 대응: 최소 구성으로 시작

## 검증 계획
- 주요 화면 동작(홈/상세/아카이브/설정) 수동 확인
- 스냅샷/스토리 실행 확인
- 번역 텍스트 누락 여부 확인
